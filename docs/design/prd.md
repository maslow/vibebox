Vibe Server PRD

## 核心用户旅程

1. **发现与注册：** 独立开发者（首要用户）发现了 Vibe Server，并使用 GitHub 或手机号快速注册了账户。
2. **订阅与支付：** 用户浏览了不同规格的套餐，选择"专业版"，并通过 Stripe 成功支付了月费。
3. **自动交付：** 后台系统监听到支付成功，自动配置了一台预装了 Vibe 环境、Web 栈 并注入了 $20 额度 API Key 的服务器。
4. **核心编码（移动优先）：** 用户在移动设备上打开原生App（或桌面端使用Web），无需任何配置，直接通过"聊天会话"界面（`happy-client`）开始与 `claude code` 对话，在通勤路上快速验证自己的 AI 创意。
5. **用量监控：** 在编码过程中，用户在移动App的仪表盘上看到自己的 API 用量从 $0.0 增长到 $3.5 / $20.0。
6. **实例管理：** 用户需要重启服务，于是在平台上点击了“重启”按钮。
7. **高级访问（Fallback）：** 用户希望安装一个新工具，于是从平台复制了 IP 和 Root 密码，通过本地 SSH 登录服务器获得了完全权限。
8. **服务支持：** 月底，用户决定暂停项目，他通过邮件联系 `support@...` 申请了取消订阅。

## 模块：用户账户与订阅管理

### 用户故事：快速注册与登录

> 作为 一个新用户（开发者/爱好者），
我希望 可以通过社交账号（GitHub/Google）或手机验证码快速注册和登录，
以便 立即开始浏览和使用平台。
> 

### 验收标准 (AC)

- [AC] 登录页面提供“使用 GitHub 登录”、“使用 Google 登录”和“手机号+验证码登录”三种方式。
- [AC] 用户使用任一方式首次登录时，系统自动为其创建账户。
- [AC] 已注册用户可以使用相同方式快速登录。

### 用户故事：浏览与选择订阅套餐

> 作为 一个潜在用户，
我希望 看到一个清晰的套餐对比页面，
以便 了解不同订阅等级（如基础版、专业版）的价格、服务器规格和包含的 API 额度。
> 

### 验收标准 (AC)

- [AC] 平台提供一个“套餐与定价”页面。
- [AC] 页面清晰展示至少两种套餐（例如“基础版”、“专业版”）。
- [AC] 每个套餐必须明确列出其核心规格、包含的 Claude API 月度额度（例如 $10/月），以及价格（月付/年付）。

### 用户故事：完成订阅支付

> 作为 一个选定套餐的用户，
我希望 通过集成的第三方支付网关（如Stripe或支付宝）完成订阅支付，
以便 激活我的 Vibe Server 实例。
> 

### 验收标准 (AC)

- [AC] 用户在套餐页点击“订阅”按钮后，被引导至 Stripe 或支付宝的标准化支付结账页面。
- [AC] 当用户支付成功后，他们被重定向回平台。
- [AC] 平台必须在用户返回时，明确显示“支付成功，实例正在配置中”的提示。

### 用户故事：安全登出

> 作为 一个已登录的用户，
我希望 可以从平台轻松登出，
以便 确保我的账户在共享设备上的安全。
> 

### 验收标准 (AC)

- [AC] 平台在用户个人资料菜单（或导航栏）的显著位置提供“登出”按钮。
- [AC] 点击“登出”后，用户的会话被立即终止，并被重定向到登录页面。

## 模块：实例交付与自动化（后台）

### 用户故事：自动化实例交付

> 作为 平台（后台系统），
当 一个新订阅支付成功时，
我需要 自动完成服务器的获取、标准环境配置 和账户绑定，
以便 用户可以“开箱即用”，无需等待人工干预。
> 

### 验收标准 (AC)

- [AC] 后台系统必须能监听到来自支付网关（Stripe/支付宝）的“支付成功”Webhook 事件。
- [AC] 监听到事件后，系统自动从服务器资源池获取一个可用实例。
- [AC] 系统自动对该实例执行标准环境安装脚本（预装 `claude code`, `happy-coder`, `Node.js`, `Python`, `Nginx`）。
- [AC] 系统自动配置与订阅等级匹配的 Claude API Key（该 Key 在 Claude 后台有额度硬顶）。
- [AC] 实例的连接信息和状态被成功绑定到发起支付的用户账户。

## 模块：Vibe Server 实例管理

### 用户故事：管理实例生命周期

> 作为 一个已订阅用户，
我希望 在我的个人面板上看到我的专属 vibe server 实例，并能对其执行 启动、停止 和 重启 操作，
以便 我可以完全控制我的开发环境状态。
> 

### 验收标准 (AC)

- [AC] 用户登录后，在“我的实例”页面可以看到一个与其订阅绑定的专属实例卡片。
- [AC] 实例卡片必须实时显示实例的当前状态（例如：运行中、已停止、启动中）。
- [AC] 平台提供功能明确的 `启动`、`停止`、`重启` 按钮。
- [AC] 执行操作时，界面提供即时的反馈（例如“正在重启中...”）。

### 用户故事：通过传统方式访问服务器（Root）

> 作为 一个高级用户，
我希望 在平台上获取我服务器的完整连接信息（IP、端口、Root凭证），
以便 我可以通过传统的 SSH 方式连接服务器，并获得完全控制权（Root Access）以进行深度定制。
> 

### 验收标准 (AC)

- [AC] 平台在实例管理界面清晰、便捷地展示服务器的 `IP 地址`。
- [AC] 平台展示 `SSH 端口` (默认为 22) 和 `Web 端口` (80/443)。
- [AC] 平台必须提供一种安全的方式，让用户能获取 `Root 密码` 或 `SSH 密钥`。
- [AC] 明确告知用户，他们拥有该实例的完全（Root）权限，可以自由安装软件或修改配置。

### 用户故事：使用预装的标准环境

> 作为 一名开发者，
当我 首次登录我的实例时，
我期望 核心的 AI 编码工具和 Web 栈（Vibe, Node, Python, Nginx）已经预装完毕，
以便 我能立即开始验证创意，而非配置环境。
> 

### 验收标准 (AC)

- [AC] 交付给用户的实例中，`claude code` 和 `happy-coder` 必须是已安装并立即可用的。
- [AC] `Node.js` 和 `Python` 的稳定版本必须已安装。
- [AC] `Nginx` 必须已安装并作为基础 Web 服务器运行。

## 模块：核心编码与 API 用量管理

### 用户故事：通过移动优先的聊天界面进行 AI 编码（核心）

> 作为 一名开发者（首要用户），
我希望 在移动App（主要）或Web端（辅助）内，直接使用一个集成的聊天界面（happy-client），
以便 我可以在任何移动场景下（通勤、咖啡厅等）与我专属服务器上的 claude code 进行对话式编码，实现最高效的原型验证。
>

### 验收标准 (AC)

- [AC] 平台提供**原生iOS/Android应用**作为主要客户端，Web端作为辅助平台。
- [AC] 移动App提供流畅的"聊天"或"Vibe"功能界面，优化触屏操作和移动场景。
- [AC] 该界面是一个深度集成的 `happy-client` 客户端（基于React Native）。
- [AC] 该客户端自动连接到用户专属的 Vibe Server 实例，无需用户手动配置 IP 或端口。
- [AC] 用户可以在移动设备上流畅地发送指令、查看文件、并接收 `claude code` 的编码响应。
- [AC] **单一统一客户端**：用户无需在不同App间切换，无需OAuth跳转，提供无缝体验。

### 用户故事：无感使用预置 API Key

> 作为 一名开发者，
当我 使用预装的 claude code 工具时，
我期望 Claude API Key 已经是预先配置好且立即可用的，
以便 我无需关心 Key 的申请、配置和泄露风险。
> 

### 验收标准 (AC)

- [AC] 实例交付时，`claude code` 已配置了与其订阅等级匹配的 API Key。
- [AC] 该 Key 在 Claude API 后台被设置了严格的月度用量上限（例如 $20），与用户订阅的额度一致。

### 用户故事：监控 API 用量

> 作为 一个已订阅用户，
我希望 在平台的显著位置（如仪表盘）清晰地看到我当前计费周期的 Claude API 用量，
以便 我能有效管理我的消耗，避免意外超支。
> 

### 验收标准 (AC)

- [AC] 平台在用户仪表盘或实例管理界面，显示一个 API 用量可视化组件（如进度条或文本）。
- [AC] 显示格式必须清晰易懂（例如：`已使用 $3.5 / $10.0 总额度`）。

### 用户故事：处理 API 额度耗尽

> 作为 一名用户，
当 我的 API 额度在本周期用尽时，
我期望 平台能明确告知我这一状态，以及额度将何时恢复，
以便 我能有清晰的预期（MVP 阶段不支持叠加包）。
> 

### 验收标准 (AC)

- [AC] 当 API 额度用尽时，实例中的 `claude code` API 调用将开始失败。
- [AC] 平台（网页/App）上的 API 用量显示区域，必须明确提示用户“额度已用尽”。
- [AC] 平台应同时告知用户：“额度将在您的下个计费周期首日自动重置”。
