## Evaluation Methodology

### Test Approach

**Scenario-Based Testing**: Simulate real Claude Code user queries and evaluate:
1. **Discoverability**: Can I find relevant docs using hashtags?
2. **Precision**: Do hashtags filter out irrelevant results?
3. **Completeness**: Are all relevant docs tagged appropriately?
4. **Redundancy**: Are there overlapping/redundant tags?
5. **Gaps**: Are critical concepts missing tags?

## Summary of Findings

### High-Value Hashtags ‚úÖ

These hashtags provide clear, unique value:

1. **`#principle:zero-modification`** - Excellent cross-document traceability
2. **`#feature:logto`** - Precise technology identification
3. **`#feature:oauth`** - Protocol-specific discovery
4. **`#component:*`** - Good component filtering
5. **`#decision:*`, `#implementation:*`, `#research:*`** - Doc type precision

### Redundant Hashtags ‚ö†Ô∏è

These may not add unique value:

1. **`#phase:*` tags** - Often redundant with doc type tags
   - `#phase:think` obvious from `#decision:*` or `#research:*`
   - `#phase:build` obvious from `#implementation:*`
   - `#phase:document` obvious from `#design:*`

   **Counter-argument**: Phase tags enable cross-type phase queries (e.g., "show all 'think' phase work regardless of type")

   **Recommendation**: Keep but use sparingly, only when cross-type phase filtering adds value

2. **`#design:system`** - May be redundant with `#design:architecture`

### Coverage Gaps ‚ùå

Missing hashtags that should exist:

1. **`docs/design/architecture.md`**:
   - Missing: `#feature:logto` (Section 7.1)
   - Missing: `#feature:oauth` (Section 7.1)
   - Missing: `#feature:authentication` (Section 7)

2. **Cross-document gaps**:
   - Some implementation docs missing `#component:client` despite client focus
   - Research docs may need more feature-specific tags

### Accuracy Issues üîß

Tags that may be misleading:

1. **`#implementation:backend`** in zero-modification-solution.md
   - Misleading: Discusses full-stack integration, not just backend
   - Better: `#implementation:full-stack` or remove this tag

## Recommendations for hashtag-curator Agent

### 1. Add Coverage Gap Detection

**Current Issue**: Agent recommended tags but missed obvious feature tags on architecture doc.

**Improvement**:
- Scan doc content for feature keywords (logto, oauth, authentication, etc.)
- Cross-reference with section headers
- Warn if major sections lack corresponding feature tags

### 2. Redundancy Detection

**Current Issue**: Recommended both doc-type tags AND phase tags, causing redundancy.

**Improvement**:
- Flag when `#phase:*` tag is redundant with doc type
- Suggest removal unless cross-type phase filtering adds value
- Provide "Keep" or "Remove" reasoning

### 3. Hierarchy Validation

**Current Issue**: Used `#component:client:web` in one doc but not consistently.

**Improvement**:
- If using hierarchical tags (`:parent:child`), enforce consistency
- Decide: Use flat tags or hierarchy? Document the convention
- Validate hierarchy depth (don't go too deep)

### 4. Content-Aware Tagging

**Current Issue**: `#implementation:backend` tag misrepresents full-stack content.

**Improvement**:
- Analyze doc content, not just title/metadata
- Count occurrences of "client", "server", "backend", "frontend" to determine scope
- Suggest more accurate scope tags

### 5. Missing Tag Inference

**Current Issue**: Didn't infer authentication-related tags from context.

**Improvement**:
- If doc has `#feature:logto` + `#feature:oauth`, likely needs `#authentication`
- If doc has `#principle:zero-modification`, check if verification docs exist and suggest linking

### 6. Quality Metrics

**Add to agent output**:
- **Coverage Score**: % of major sections with relevant tags
- **Redundancy Score**: % of tags that are redundant
- **Precision Score**: % of tags that accurately represent content
- **Discoverability Score**: Can AI find this doc via relevant queries?
